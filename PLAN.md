День 2: Модели данных (База)
Задача: Создать таблицы в Postgres.
Модель User:
id (int, PK)
telegram_id (bigint, unique)
username (string)
created_at (datetime)
Модель Payment:
id (int, PK)
user_id (FK -> User)
amount (decimal)
currency (string, default 'RUB')
status (string: 'pending', 'success', 'failed')
transaction_id (string, от Lava)
created_at (datetime)
Модель Subscription:
id (int, PK)
user_id (FK -> User)
plan_id (string: 'basic', 'pro')
expires_at (datetime)
is_active (boolean)
Миграции:
Настроить Alembic (или создать таблицы вручную через SQL для скорости на старте).
День 3: Lava Интеграция
Задача: Научиться создавать платежи.
Сервис LavaService:
Метод create_payment(amount, user_id, description):
Формирует подпись (HMAC SHA256) через секретный ключ.
Отправляет POST запрос на API Lava.
Возвращает payment_url.
Метод verify_signature(headers, body):
Проверяет, что вебхук пришел действительно от Lava.
Логирование:
Все запросы к Lava писать в файл lava.log (для разборов полетов).
День 4: Webhook (Обработка оплаты)
Задача: Автоматически активировать доступ после оплаты.
Эндпоинт: POST /api/webhook/lava.
Логика:
Проверка подписи (Security).
Проверка идемпотентности (чтобы не начислить подписку дважды за один платеж).
Если статус success:
Обновить Payment.status.
Найти или создать Subscription.
Продлить expires_at (например, +30 дней).
Отправить событие в бот.
Тест: Использовать тестовый режим Lava (sandbox).
День 5: Telegram-бот шаблон
Задача: Интерфейс для пользователя.
Библиотека: aiogram 3.x.
Команды:
/start — Приветствие, проверка подписки.
/pay — Показать кнопки тарифов.
/status — Показать дату окончания подписки.
Клавиатуры:
Inline-кнопки с ссылками на оплату (генерируются через API).
Middleware (Проверка доступа):
Перед выдачей контента проверять: if subscription.is_active and subscription.expires_at > now.
Если нет — кнопка «Оплатить».
День 6: Тест полного цикла
Задача: Убедиться, что все работает вместе.
Сценарий 1 (Успех):
Нажать «Оплатить» → Перейти по ссылке → Оплатить (тестовой картой) → Получить сообщение «Оплата прошла» → Доступ открыт.
Сценарий 2 (Истечение):
Вручную в базе изменить expires_at на прошлое → Зайти в бот → Доступ закрыт.
Сценарий 3 (Ошибка):
Отменить платеж → Получить сообщение «Платеж отменен».
Результат: Готовый репозиторий telegram-payment-template.

ПРАВИЛА БЕЗОПАСНОСТИ (ДЕТАЛИЗАЦИЯ)
1. Telegram-аутрич
Лимит: Строго не больше 15 новых диалогов в сутки с одного аккаунта.
Аккаунты: Иметь 2 запасных аккаунта (на разные сим-карты), если основной уйдет в спам-блок.
Персонализация: Всегда начинать с имени или названия проекта. Никакого «Здравствуйте, у меня есть предложение».
2. Поддержка
Границы: В договоре/чате закрепить: «Поддержка включает исправление багов. Новые функции — отдельная оплата».
Срок ответа: «Отвечаю в рабочее время 10:00–19:00 МСК». Не будь доступен 24/7.
3. Юридическая часть
Текст «Условия работы»: Сохрани как шаблон сообщения. Отправляй перед началом работ.
Предоплата: Работа начинается только после скрина/чека о переводе 50%.
ТЗ: Все договоренности фиксировать текстом в чате. «Правильно ли я понял, что делаем Х?». Клиент должен ответить «Да».
4. Хостинг и Сервер
Контроль: Лучше арендовать сервер на себя (VPS), а клиенту выставлять счет с наценкой.
Ты платишь провайдеру $5.
Клиент платит тебе $10.
Ты имеешь доступ к серверу для фикса проблем, клиент не сломает конфигурацию.
Бэкапы: Настроить автоматический дамп базы данных раз в сутки на удаленное хранилище (S3 или другой сервер).
5. Безопасность кода
Секреты: Никогда не коммить .env в git.
Ключи: Не передавать ключи API в открытом виде. Использовать переменные окружения.
Доступ: SSH доступ к серверу только у тебя (или через ключи, если клиент технарь).
